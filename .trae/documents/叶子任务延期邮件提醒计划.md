## 二、功能设计

### 2.1 延期判断逻辑

```
任务延期条件：
  (任务结束日期 < 当前日期) AND (任务状态 != "done")

叶子任务判断：
  该任务没有子任务（isLeafTask() 返回 true）

如果没有叶子任务，则检查父任务本身
```

### 2.2 通知对象

| 角色    | 来源                          | 说明    |
| ----- | --------------------------- | ----- |
| 任务负责人 | task.assigneeId             | 直接接收人 |
| 项目负责人 | project.ownerId             | 抄送接收人 |
| 项目经理  | user.role = "manager" 或类似角色 | 可选抄送  |

### 2.3 提醒频率

* **每日提醒**：每天上午 9:00 检查并发送提醒

* **避免重复**：同一任务 24 小时内只提醒一次

## 四、核心逻辑详细说明

### 4.1 延期任务扫描流程

```
1. 获取所有状态为 "active" 或 "planning" 的项目
2. 对每个项目：
   a. 获取该项目的所有任务
   b. 筛选出延期任务：endDate < today AND status != "done"
   c. 对每个延期任务：
      i. 判断是否为叶子任务
      ii. 如果是叶子任务 → 处理
      iii. 如果不是叶子任务，检查是否有叶子子任务延期
           - 如果有 → 跳过（叶子任务会单独提醒）
           - 如果没有 → 将该父任务视为叶子任务处理
   d. 收集所有需要提醒的任务
3. 对每个需要提醒的任务：
   a. 获取任务负责人
   b. 获取项目负责人
   c. 获取项目经理列表
   d. 检查过去24小时内是否已提醒过该用户
   e. 如果未提醒 → 发送邮件
   f. 记录提醒
```

### 4.2 避免重复提醒的逻辑

```
查询条件：
- task_id = 当前任务ID
- notified_user_id = 接收人ID
- notification_date >= 昨天

如果查询结果为空 → 可以发送
否则 → 跳过
```

### 新增表：delay\_notification\_record

```sql
CREATE TABLE delay_notification_record (
    id VARCHAR(50) PRIMARY KEY,
    task_id VARCHAR(50) NOT NULL,
    project_id VARCHAR(50) NOT NULL,
    notified_user_id VARCHAR(50) NOT NULL,
    notified_email VARCHAR(255) NOT NULL,

    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_task_date (task_id, notified_user_id, notification_date),
    INDEX idx_project_id (project_id)
```

## 八、注意事项

1. **任务状态**: 只对非 "done" 状态的任务进行延期检查
2. **邮件频率**: 同一任务给同一用户 24 小时内只提醒一次
3. **性能考虑**: 大量任务时考虑分批处理
4. **异常处理**: 邮件发送失败不应影响后续任务处理
5. **日志记录**: 详细记录每次检查和发送的日志
6. **时区处理**: 使用系统配置的时区（Asia/Shanghai）
